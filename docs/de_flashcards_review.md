### 1 - Programming

#### Conventional Commits

*Git / DevOps*

A specification for commit messages with structured prefixes (feat:, fix:, chore:, docs:, refactor:, etc.). Enables automated changelog generation, semantic versioning, and easy filtering of git history. Widely adopted in modern open-source and enterprise codebases.

#### Parquet

*Data Formats*

A columnar file format widely used in analytics. Stores data by column rather than by row, enabling aggressive compression and fast column-selective reads. Supports predicate pushdown and is the de facto standard for data lake storage.

#### Pre-commit Audit Discipline

*Git / DevOps*

Always run `git status` and `git diff --cached` before committing. Knowing what you're about to commit prevents accidental secret leaks, includes-everything-by-mistake commits, and 'WTF did I commit' moments later. Counters the bad habit of `git add . && git commit -am 'stuff'`.

### 10 - Systems

#### Configuration Cascade Pattern

*DevOps*

Configuration values flow through layers: file → tool → process → consumer. When tracing where a value 'comes from', walk the chain backward to find which layer dropped or modified it. Same pattern recurs across dbt profiles, Terraform vars, K8s ConfigMaps, Airflow connections — naming the layer makes debugging deterministic.

### 3 - Modeling

#### Medallion Architecture

*Data Modeling*

A data lakehouse/warehouse design pattern organizing data into three quality tiers: Bronze (raw, as-ingested), Silver (cleaned, conformed, typed), and Gold (business-ready, aggregated, modeled). Maps to RAW / STAGING / MARTS schemas in many dbt projects.

### 4 - Warehousing

#### ACCOUNTADMIN

*Snowflake*

Snowflake's most-privileged built-in role. Required for creating storage integrations, managing account-level settings, and bootstrap operations. Should never be used for routine work; create dedicated roles instead. Treat similar to AWS root or Linux root.

#### AUTO_RESUME

*Snowflake*

A Snowflake warehouse parameter that, when TRUE, automatically restarts a suspended warehouse when a query is submitted to it. Resume latency is typically ~1 second. Paired with AUTO_SUSPEND for hands-off cost management.

#### AUTO_SUSPEND

*Snowflake*

A Snowflake warehouse parameter (in seconds) that automatically pauses the warehouse after a period of inactivity. Critical cost-control setting. AUTO_SUSPEND = 60 means the warehouse stops billing 60 seconds after the last query completes.

#### COPY INTO

*Snowflake*

Snowflake SQL command for bulk loading data from a stage into a table, or unloading from a table to a stage. Supports file formats like CSV, JSON, Parquet, Avro. Core primitive for ELT ingestion.

#### Functional Role vs Access Role

*Snowflake*

Snowflake-recommended pattern for scalable RBAC. Access roles bundle privileges on specific objects (e.g., `MARTS_READ`, `MARTS_WRITE`). Functional roles represent job functions (e.g., `ANALYST`, `DATA_ENGINEER`) and are composed by being granted access roles. Users are granted functional roles. This separates 'what can be done' from 'who does it' for easier permission management at scale.

#### GRANT ON FUTURE

*Snowflake*

Snowflake feature that automatically grants privileges on objects (tables, views, etc.) created in the future within a schema or database. Eliminates the need to re-grant after every new object creation. Example: `GRANT SELECT ON FUTURE TABLES IN SCHEMA X TO ROLE Y`.

#### GRANT ROLE

*Snowflake*

Snowflake DDL that gives a user (or another role) the ability to assume a role via USE ROLE. Without this grant, even the role's creator cannot use it. The GRANT must be issued from a role with sufficient privileges (typically ACCOUNTADMIN or SECURITYADMIN).

#### LIST (stage command)

*Snowflake*

Snowflake SQL command that lists files in an external or internal stage, including size and last-modified timestamps. Used to verify stage connectivity and inspect what files are available for COPY INTO. Example: `LIST @NYC_TAXI.RAW.NYC_TAXI_STAGE/path/`.

#### Micro-partition

*Snowflake*

Snowflake's internal storage unit: immutable, compressed, columnar files of 50–500 MB (uncompressed) that hold rows of a table. Snowflake automatically manages partitioning based on ingestion order; queries prune partitions using column metadata for fast scans.

#### Snowflake External Stage

*Snowflake*

A named Snowflake object pointing to a location in external cloud storage (S3, GCS, or Azure Blob). Referenced by COPY INTO commands for bulk loading/unloading. Can use a storage integration or (less securely) embedded credentials.

#### Snowflake Object Namespace

*Snowflake*

Every Snowflake object lives in a three-level namespace: `database.schema.object`. Unqualified references (e.g., `MY_TABLE`) resolve against the session's current database and schema. Interactive queries can rely on context, but automated scripts and dbt models should always fully-qualify to avoid resolution bugs.

#### Snowflake RBAC Model

*Snowflake*

Snowflake's access control model. Users authenticate; roles authorize. A user cannot use a role unless granted via `GRANT ROLE <role> TO USER <user>`. Permissions attach to roles, not users. Roles can be granted to other roles, creating role hierarchies for inheritance.

#### Snowflake Storage Integration

*Snowflake*

A Snowflake object that encapsulates an external cloud IAM role (e.g., an AWS IAM role) for accessing external storage. Snowflake assumes the role via STS to generate temporary credentials for each request. Preferred over embedding access keys in stage DDL because credentials rotate automatically and never appear in Snowflake metadata.

#### Snowflake Virtual Warehouse

*Snowflake*

A cluster of compute resources in Snowflake that executes queries. Decoupled from storage — can be sized, paused, and resumed independently. Sizes scale in powers of two (XS, S, M, L, XL, ...), with each size doubling cost and compute.

#### Snowflake Worksheet

*Snowflake*

Snowflake's web UI tool for interactive SQL development. Critical gotcha: by default, the Run button executes only the statement at cursor position, not the entire script. To run multiple statements, highlight all text (Cmd+A) before clicking Run, or use the Run All option.

#### STORAGE_ALLOWED_LOCATIONS

*Snowflake*

A defense-in-depth setting on Snowflake storage integrations that restricts which S3 (or GCS/Azure) paths the integration may access. Evaluated by Snowflake *before* any AWS call. Prevents an over-permissioned IAM role from being misused via Snowflake. Even if the IAM role grants access to a bucket, Snowflake won't allow the integration to use it unless explicitly listed.

#### USE ROLE

*Snowflake*

Snowflake session command that switches the active role for the current session. Affects which objects the user can see and operate on. Returns an error if the user has not been granted the role.

### 5 - Pipelines

#### Airflow Executor

*Orchestration*

The component that determines *how* Airflow runs tasks. SequentialExecutor (SQLite, one task at a time, dev only), LocalExecutor (parallel on one machine, Postgres/MySQL metadata DB), CeleryExecutor (distributed across worker nodes), KubernetesExecutor (one pod per task).

#### Airflow Metadata Database

*Orchestration*

The relational database (typically Postgres) that stores Airflow's state: DAG definitions, task instances, run history, user accounts, connections, variables, XComs. Critical infrastructure — losing this means losing all run history. Backed up regularly in production.

#### Airflow Task Execution Patterns

*Orchestration*

Airflow can either *execute* tasks in its own Python environment or *remote-control* them by triggering external systems. Production prefers remote-control: dbt runs in its own image, Spark runs on its own cluster, etc. Operators like KubernetesPodOperator, DockerOperator, or HTTP calls trigger work externally. Avoids dependency conflicts and lets each tool's stack evolve independently.

#### Custom Airflow Dockerfile

*Orchestration*

Production Airflow deployments use a custom Dockerfile extending apache/airflow:<version> with required Python packages baked in at build time. This makes container starts fast and reproducible. Contrast with `_PIP_ADDITIONAL_REQUIREMENTS`, which installs packages on every container start — convenient for dev, unsuitable for production.

#### DAG

*Orchestration*

Directed Acyclic Graph. In Airflow (and most orchestrators), a DAG is a collection of tasks with dependencies expressed as edges. 'Acyclic' means no task can depend on itself directly or transitively — guarantees the graph can be topologically sorted and executed.

#### ELT vs ETL

*Data Pipelines*

ETL = Extract, Transform, Load: transformation happens in an intermediate compute layer before loading into the warehouse. ELT = Extract, Load, Transform: raw data is loaded into the warehouse first, then transformed using the warehouse's compute. ELT dominates modern stacks because cloud warehouses (Snowflake, BigQuery) are cheap and powerful enough to transform at scale.

#### Idempotency

*Data Pipelines*

A property of an operation such that running it multiple times produces the same result as running it once. Critical for data pipelines: if a task retries due to failure, it should not duplicate rows or cause inconsistent state. Common techniques: MERGE instead of INSERT, partition-replace, deterministic primary keys.

#### LocalExecutor

*Orchestration*

Airflow executor that runs tasks as parallel subprocesses on a single machine. Requires a metadata database that supports concurrency (Postgres, MySQL — not SQLite). Suitable for development and small-scale production. Steps up to CeleryExecutor or KubernetesExecutor for distributed workloads.

### 7 - Cloud

#### AssumeRole

*AWS / IAM*

The STS API call that lets one principal temporarily take on the permissions of an IAM role. Returns a triple of temporary access key ID, secret access key, and session token. Snowflake uses this under the hood for storage integrations.

#### AssumeRole as Risk Transfer

*Security / Architecture*

AssumeRole isn't just about token expiration — it's an architectural pattern that prevents a third-party service (like Snowflake) from having to STORE customer credentials. With long-lived keys, Snowflake would carry breach risk for thousands of customers' AWS accounts. With AssumeRole, Snowflake stores zero customer credentials; a breach yields nothing useful. The keystore problem is eliminated entirely.

#### AWS STS

*AWS / IAM*

AWS Security Token Service. An AWS service that issues temporary, short-lived credentials (typically 15 minutes to 12 hours) in response to requests like AssumeRole, GetSessionToken, and GetFederationToken.

#### boto3 Credential Chain

*AWS*

Order in which boto3 searches for AWS credentials when making API calls: (1) environment variables AWS_ACCESS_KEY_ID/AWS_SECRET_ACCESS_KEY, (2) shared credentials file ~/.aws/credentials, (3) IAM role attached to instance (EC2/ECS/Lambda). Stops at first match. Determines WHICH AWS identity authenticates from a given execution context.

#### CloudTrail

*AWS*

AWS service that logs every API call made within an account. Critical debugging tool for cross-service auth failures: shows AssumeRole events with reason codes, S3 data access (with data event logging enabled), and SCP/IAM policy denials. First place to look when an AWS call fails with a generic AccessDenied — CloudTrail typically explains why.

#### Confused Deputy Problem

*Security*

A security vulnerability where a privileged service is tricked into using its access on behalf of an unauthorized attacker. Mitigated in AWS via the ExternalId condition in role trust policies, ensuring only the intended caller can assume the role.

#### ExternalId

*AWS / IAM*

A string value included in a role's trust policy Condition block to prevent the 'confused deputy' problem. When external services assume the role, they must present the matching ExternalId, ensuring only the intended caller (not just anyone with the role ARN) can assume it.

#### IAM Policy

*AWS / IAM*

A JSON document attached to an IAM identity (user, group, or role) or to a resource, declaring which actions are Allowed or Denied on which resources. Evaluated on every API call.

#### IAM Role

*AWS / IAM*

An AWS identity with permissions policies attached, but no long-term credentials. Unlike an IAM user, a role is *assumed* by trusted principals (users, services, other accounts) who receive temporary credentials. Roles are the preferred mechanism for cross-service trust.

#### IAM User

*AWS / IAM*

An AWS identity that represents a person or long-lived application, with permanent credentials (password and/or access keys). Less secure than roles for service-to-service auth because credentials don't rotate automatically.

#### Managed Policy vs Inline Policy

*AWS / IAM*

Managed policies are reusable AWS-maintained or customer-created permission sets that can attach to many identities. Inline policies are embedded directly into one specific identity (user, group, or role). Use managed for shared permissions across many identities; use inline for one-off, identity-specific permissions like a single service role.

#### Principle of Least Privilege

*Security*

A security principle stating that any identity should have only the minimum permissions required to perform its task, scoped to only the resources it needs, for only as long as it needs them. Central to good IAM design.

#### S3 Bucket

*AWS / Storage*

A top-level container in Amazon S3 for objects (files). Bucket names are globally unique across all AWS accounts. Buckets live in a specific AWS region; cross-region data transfer incurs cost.

#### S3 Object

*AWS / Storage*

An individual file stored in an S3 bucket, identified by a key (the full path within the bucket). Objects are immutable — 'updating' actually creates a new version (with versioning enabled) or overwrites the existing key.

#### System Cast vs Runtime Cast

*Security / Architecture*

Distinction between identities that EXIST in a system (built it, configured it, can use it) and identities that ACT during a specific execution. When asked 'who acts during X', filter out setup-only identities (root admin, ACCOUNTADMIN, human users) — they're system cast. The runtime cast is typically narrower and includes orchestration components, the script's outbound auth identities, and any cross-cloud assumed roles.

#### Trust Policy

*AWS / IAM*

A special IAM policy attached to a role that declares *who can assume the role*. Defines the Principal (the identity allowed to assume it) and optional Conditions (e.g., ExternalId). Distinct from permissions policies, which declare what the role can do once assumed.

#### Trust Policy vs Permissions Policy

*AWS / IAM*

Two distinct IAM policies attached to a role. Trust policy authorizes WHO can assume the role (defines Principal and Conditions). Permissions policy authorizes WHAT the assumed role can do once assumed (defines Actions on Resources). Both must succeed for an action to complete. Many AccessDenied errors stem from a correct trust policy but wrong permissions policy.

### 8 - Infra

#### Bind Mount vs Volume (Docker)

*DevOps*

Both expose host data to containers, but differ in management. Bind mount: maps a specific host path into the container; tied to host filesystem. Volume: managed by Docker, portable across hosts, survives container removal. Volumes preferred for persistent data (databases); bind mounts useful for live-editing source code during development.

#### Compose Variable Substitution

*DevOps*

Docker Compose auto-loads .env from the directory you run `docker compose up` in (Compose convention, not shell convention) and substitutes ${VAR} or ${VAR:-default} references in YAML at compose-load time. Your shell cannot see these variables via `echo $VAR` — different processes read from different sources.

#### Docker Daemon vs Client

*DevOps*

The `docker` CLI command is a client that talks to the Docker daemon (a long-running background process). Operations like pulls and builds happen in the daemon, not the client. Ctrl+C kills the client but may leave daemon state in-flight or locked, causing later operations to hang. Restarting Docker Desktop is often the fastest fix for stuck pulls or builds.

#### Docker Image Layers

*DevOps*

Docker images are built as ordered, immutable layers. Each Dockerfile instruction creates a layer. Layers are content-addressed (hashed) and cached locally, so common base layers are downloaded once and reused across images. This is why a first pull of an image is slow but subsequent pulls of related images are fast.

#### Docker Image vs Container

*DevOps*

An image is an immutable, layered template (like a class). A container is a running instance of an image (like an object). Images are built from a Dockerfile; containers are started from images via `docker run` (or `docker compose up`).

#### Docker Volume

*DevOps*

A persistent storage mechanism for containers, managed by Docker independently of the container lifecycle. Unlike bind mounts (host-path-dependent), volumes are portable across hosts and survive container removal. Used in our Airflow stack for Postgres metadata persistence.

#### macOS Application PATH

*DevOps*

`.app` bundles on macOS are directories (not single files) and their contents are not on the shell's PATH by default. Tools shipped inside `.app` bundles (like Docker's CLI) require either explicit PATH entries pointing into the bundle, or symlinks in standard locations like /usr/local/bin.

#### Tier 2 Configuration (Homebrew)

*DevOps*

A Homebrew support tier indicating that pre-compiled binaries (bottles) are not provided for the platform. Installs require local compilation from source, which can take 30+ minutes for large dependencies like Python. Common on older Intel Macs running recent Homebrew. Workaround: use vendor-provided installers (e.g., AWS .pkg) when available.

### 1 - Programming

#### Naming Convention as Automation Surface

*Data Engineering / DevOps*

Predictable naming patterns ({type}_tripdata_{YYYY}-{MM}.parquet) let one function or query handle the entire family of artifacts. Unpredictable naming forces per-instance special-case logic. Naming conventions are not bureaucracy — they are design contracts that future automation depends on. Apply to file paths, S3 prefixes, table names, DAG IDs, dbt models.

### 2 - SQL/DBs

#### SQL GROUP BY Rule

*SQL*

In strict SQL engines (DuckDB, Postgres, Snowflake), every column in SELECT must either appear in GROUP BY or be wrapped in an aggregate function. MySQL historically allowed loose grouping (silently picking arbitrary values), which led to subtle bugs. Errors here usually mean the query's intent is logically ambiguous, not just syntactically wrong. The strict behavior forces precise articulation of intent.

#### TRY_CAST vs CAST

*SQL*

CAST throws an error and fails the row when source data can't be converted to target type. TRY_CAST returns NULL instead. For exploratory or historical data with type drift across years, TRY_CAST is the safer default — failed conversions become observable as NULLs rather than blocking the entire load. Use TRY_CAST in STAGING dbt models where source types may vary; reserve CAST for cases where a failure SHOULD block (e.g., critical PK columns).

#### Type Coercion Safety

*SQL / Data Engineering*

Widening type casts (INTEGER→DOUBLE, INT→BIGINT) are automatic and lossless. Narrowing casts (DOUBLE→INTEGER, BIGINT→INT) can lose data or fail — verify value ranges first. DOUBLE→BIGINT requires that all values are integer-shaped (no fractional component); query for fractional values before assuming the cast is safe. The principle: trust your cast direction instinct, then verify with a query.

### 3 - Data Modeling

#### Degenerate Dimension

*Data Modeling*

A categorical attribute that's logically dimensional (a code or category) but small/simple enough to leave inline in the fact table without a separate lookup. Fails one of the three richness tests (cardinality, stability, descriptive richness). Examples: store_and_fwd_flag (Y/N), is_gift (boolean), passenger_count (1-6). No JOIN needed; the value is already self-explanatory.

#### Deterministic vs Non-Deterministic Dimensions

*Data Modeling*

Deterministic: a dimension whose values can be GENERATED from rules (e.g., dim_date generated by code; dim_time for hour/minute analysis). No external source needed. Non-deterministic: a dimension whose values must be INGESTED from a source of truth (e.g., dim_location from TLC's zone CSV; dim_customer from CRM). Generation suits the former; ingestion pipelines suit the latter.

#### Dimension Table

*Data Modeling*

A warehouse table that provides context for fact rows — the WHO, WHAT, WHERE, WHEN, HOW. Each row represents an entity (a customer, location, product, date) with descriptive attributes. Smaller than fact tables (hundreds to millions of rows). Connected to facts via foreign keys; provides human-readable values via JOIN at query time.

#### DuckDB

*Tooling*

In-process columnar SQL engine — conceptually 'SQLite for analytics.' Queries Parquet/CSV/JSON files directly without loading them into memory. Streams data through the query engine in chunks, so it can profile files larger than RAM. SQL syntax is highly compatible with Snowflake/BigQuery/Postgres, making it a natural learning bridge to production warehouses. Free, embedded, no server.

#### Encoding Drift vs Definition Drift

*Data Engineering / Profiling*

Two structurally different categories of semantic drift with different fixes. Encoding drift: the measurement system changed (USD vs EUR; cents vs dollars; UTC vs local time). Same real-world concept, different unit. Fix: normalize unit using historical conversion data. Definition drift: the real-world concept itself changed (revenue includes refunds vs excludes; ad-impression requires viewability vs not). Same number, different meaning. Fix: segment the analysis — pre-change and post-change periods cannot be combined; compare like-for-like within each definition era.

#### Fact Table

*Data Modeling*

A warehouse table that records events or measurements — things that happened. Each row represents one event at the table's grain (e.g., one taxi trip). Contains foreign keys to dimensions, degenerate dimensions, and pure measures. Typically the largest table in a star schema by row count.

#### Grain

*Data Modeling*

The precise definition of what one row in a fact table represents — the unit of analysis. Stated as 'one row per X' (e.g., one row per completed taxi trip). Determines what questions are answerable: too coarse loses detail; too fine wastes storage. The most important decision in fact-table design — many schema problems trace back to fuzzy grain.

#### Predict Before You Query

*Data Engineering / Profiling*

A profiling discipline: before running an exploratory query, write down what you EXPECT the result to be. The gap between prediction and reality is where insight lives — far more valuable than the query result alone. Predicting forces you to articulate your mental model; gaps reveal hidden assumptions. Apply for any unfamiliar dataset, data quality investigation, or upstream-change validation.

#### Richness Test

*Data Modeling*

Three-criteria heuristic for deciding if a column should become a dimension table: (1) cardinality > 2-5, (2) values may grow or change over time, (3) plausible to attach descriptive attributes. Failing any one pushes toward degenerate. Shortcut: 'Would a business analyst writing a report ever want to JOIN to this column for richer text?' Yes → dimension; No → degenerate.

#### Role-Playing Dimension

*Data Modeling*

The same physical dimension table referenced multiple times by one fact table, with each reference playing a different role (e.g., dim_date as both pickup_date and dropoff_date; dim_location as both pickup and dropoff). SQL queries use aliases to disambiguate roles. Avoids duplicating physical dimension data while supporting multiple semantic relationships.

#### SCD Type 0

*Data Modeling*

Slowly Changing Dimension where attributes are FIXED once defined and never change. Calendar dimensions (dim_date) are typical Type 0 — once Jan 15 2024 is defined as a Monday, it stays that way forever. New rows can be added (extending the range), but existing rows are immutable. Simplest type; no change-tracking machinery required.

#### SCD Type 1

*Data Modeling*

Slowly Changing Dimension where attribute changes OVERWRITE the existing row. History is lost. Appropriate for cosmetic changes (typo fixes, label updates) where the semantic meaning hasn't changed. Cheap and simple. Example: payment_type description goes from 'Cash' to 'Cash Tender' — the meaning didn't change, only the label.

#### SCD Type 2

*Data Modeling*

Slowly Changing Dimension where attribute changes create a NEW ROW with valid_from/valid_to dates, preserving historical versions. Each version has its own surrogate key. Fact tables JOIN via surrogate so each fact row matches the dimension version active at the event time. Standard for analytical warehouses needing historical fidelity. Storage and query complexity cost; durable history benefit.

#### SCD Type 3

*Data Modeling*

Slowly Changing Dimension that adds a column for the previous attribute value (e.g., current_zone, previous_zone). Limited history — only the last change is preserved. Rare in practice; useful mainly for transitional reporting where stakeholders want to see one prior name during a rename rollout. For multi-step history, Type 2 is the standard choice instead.

#### SCD2 + is_current Filter Trap

*SQL / Data Modeling*

Filtering 'WHERE dim.is_current = TRUE' on a fact-to-Type-2-dimension join silently drops historical fact rows whose linked dimension version has been superseded. The bug returns results without errors but with artificially low aggregate values — and specifically hits long-tenured entities with attribute changes. Fix: omit the filter; SCD2 surrogate keys attribute facts to versions active at the event time automatically.

#### Schema vs Semantics Stability

*Data Engineering / Profiling*

Schema (column names, types, presence) and semantics (whether the same recorded value represents the same real-world thing across history) can drift independently across time. A column may exist in the schema from day one but be unpopulated until a policy launches; values may stay numerically identical while their unit, definition, or business meaning shifts beneath them. Schema checks (DESCRIBE) are fast; semantic checks (querying value distributions, units, definitions per period) are slower but essential. Five drift categories: NULL-population, unit/encoding, definition, taxonomy expansion, soft-state.

#### Snowflake Schema

*Data Modeling*

A dimensional design where dimensions are normalized into multiple sub-dimension tables connected by foreign keys (e.g., dim_location → dim_borough → dim_region). Multi-hop joins required for some attributes. Historically used to save storage; columnar compression has largely eliminated that benefit. Rare in modern greenfield analytical warehouses; query simplicity of star usually wins.

#### Star Schema

*Data Modeling*

A dimensional design where the fact table sits at the center with dimension tables radiating outward, each one join away. Named for its star-like shape when diagrammed. Modern columnar warehouses prefer star over normalized alternatives because compression handles repeated dimension values efficiently, while query simplicity from one-hop joins is meaningful.

#### Structural Pass vs Semantic Pass

*Data Engineering / Profiling*

The two-stage data validation discipline. Structural pass: fast checks against schema (column count, names, types). Semantic pass: slow checks against actual data (value distributions per period, unit consistency, definition stability). The classic failure mode is using the fast structural pass as a proxy for the slow semantic pass — schema looks identical, analyst skips semantic verification, analysis returns plausible-looking but quietly wrong numbers. The danger isn't crashes; it's silent wrongness. Always run both passes for any historical dataset before trusting cross-period analysis.

#### Surrogate Key vs Natural Key

*Data Modeling*

Natural key: the identifier from the source system that carries business meaning (e.g., location_id = 161 in TLC data). Surrogate key: a warehouse-internal row identifier with no business meaning (e.g., location_sk = 1001). SCD Type 2 REQUIRES surrogate keys because natural keys appear in multiple rows (one per version). Many warehouses use surrogates universally for consistency and source-system independence.

### 4 - Warehouse

#### Snowflake CLUSTER BY

*SQL / Performance*

A hint to Snowflake's optimizer about micro-partition organization. ALTER TABLE x CLUSTER BY (date_column) lets queries filtering on that column skip entire micro-partitions, dramatically reducing scan time. Cost: ongoing automatic reclustering as data is loaded. Worth it for tables >1B rows or with frequent date-range queries. Different from the older 'CLUSTER BY' in standard SQL — Snowflake-specific feature.

### 8 - Infra

#### Sync Tools and Dev Projects Don't Mix

*DevOps / Workflow*

Cloud sync tools (OneDrive, Dropbox, iCloud Drive, Google Drive) actively watch managed folders for changes and trigger uploads on every modification. This conflicts with rapid file operations from git, pip, dbt, Docker, etc. — causing slowdowns, partial writes, lock contention, and inadvertent uploads of gitignored secrets. Dev projects belong on a local filesystem (e.g., ~/Projects/) that no sync tool watches.

#### WSL2 9P Translation Layer

*DevOps / WSL2*

WSL2 mounts the Windows filesystem at /mnt/c/... via the 9P protocol. Most operations work, but rm, chmod, chown, and other permission-changing ops can fail when Windows-side ACLs or sync agents (OneDrive, Dropbox) lock the file. Solution: perform such operations on the Windows side, or keep dev projects on the Linux filesystem (/home/...) entirely.
